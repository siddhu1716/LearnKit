LearnKit Documentation

LearnKit is an agent-agnostic SDK that gives any AI agent a self-improving memory layer through experience distillation — "fine-tuning without fine-tuning." Performance improves over time while the underlying model weights never change.

Overview

Unlike traditional memory systems that store raw chat logs, LearnKit analyzes agent execution traces to extract structured, reusable artifacts — successful workflows, verified facts, and known failure patterns. These distilled artifacts are injected back into the agent's context on future runs, so the agent compounds knowledge instead of starting from scratch each time.

  • Agent-agnostic — wraps any agent with a single decorator.
  • Typed & auditable — every record is a structured, scored JSON entry.
  • Bounded context — retrieval is token-capped (max ~1,200 tokens), never "memory soup."
  • No retraining — zero GPU hours, no model changes, no vendor lock-in.

Getting started

Install the package from PyPI:

# Python 3.10+
pip install learnkit-ai

Wrap your agent with the @lk.agent decorator and run it as normal — LearnKit handles classification, retrieval, and distillation in the background.

import learnkit
lk = learnkit.LearnKit()

@lk.agent(domain="legal")
def agent(task):
    return llm.run(task)

# Run 1:  Figures it out from scratch
# Run 10: Has distilled workflows from 9 prior runs
# Run 50: Has a curated wiki of domain expertise

Local-first. By default, records persist to a local SQLite store. No external services are required to start.

Core philosophy: experience distillation

The central tenet of LearnKit is that distilled experience is superior to raw memory. Raw chat logs create "memory soup" — exploding context windows with a high noise-to-signal ratio. LearnKit replaces this with a curated wiki of expertise.

Raw memory storageLearnKit distillation
Storage contentRaw chat logs / full historyStructured, typed records
Context managementGrows linearly; "memory soup"Bounded, curated, ~1,200-token cap
Quality controlNone; stores everythingLLM-as-judge scores every run
ReusabilityHard to generalizeCross-domain via task classification
Failure handlingRepeats mistakesFailures are first-class, used for avoidance

The 8-step loop

When an agent is decorated with @lk.agent, LearnKit orchestrates a continuous loop of classification, retrieval, execution, and distillation around your existing logic.

  1. Classify

    TaskClassifier uses DSPy to generate a domain vector and task type.

  2. Route

    MemoryRouter selects which memory stores and record types are relevant.

  3. Retrieve

    SemanticRetriever runs hybrid search (BM25 + vector) for the top ~8 records.

  4. Compose

    ContextComposer compresses matches into a prompt block (~1,200-token cap).

  5. Execute

    Your agent function runs with the distilled context injected.

  6. Evaluate

    Evaluator (LLM-as-judge) assigns a quality score from 0–5.

  7. Distill

    If the score clears the gate, MemoryDistiller extracts reusable artifacts.

  8. Store

    SQLiteBackend persists the new records for future runs.

Quality gate. Only runs scoring above the quality_threshold (default 3.5) produce skills or facts. Failures are stored immediately to prevent repeating dead-end paths.

Key subsystems

Ingest

The Distiller

Analyzes the Trajectory after each task. Successes become Skill/Fact records; failures become Failure records.

Query

The Retriever

Hybrid BM25 + vector search returns the top 8 records (or 1,200 tokens) of relevant experience before execution.

Maintain

The Evolver

Confidence decay, quarantine promotion, and GEPA (Genetic-Pareto Prompt Evolution) keep memory curated.

The 7 memory types

Every distilled artifact is one of seven typed records, each with its own schema and retrieval behavior.

Skill

A reusable, step-by-step procedure proven to work on a class of tasks.

Fact

A verified statement of truth, optionally with temporal validity that can expire.

Failure

A known dead-end or mistake, surfaced immediately to steer the agent away.

Strategy

A higher-level approach or plan for tackling a category of problems.

Preference

A user- or domain-specific convention the agent should honor.

Heuristic

A rule-of-thumb that biases decisions without guaranteeing an outcome.

Trace

A captured execution trajectory used for replay and procedure induction.

Memory lifecycle

Records move through three states, managed by the maintain_memory() routine:

  • Quarantine — newly distilled records are held on probation before they can influence runs.
  • Active — promoted records that are eligible for retrieval and injection.
  • Stale — records whose confidence has decayed from disuse; eventually pruned.

Self-curating. Memory is bounded by design — confidence decays over time, high-value skills are optimized via GEPA, and low-performers stale out automatically.

Two learning paths

Pick the decorator that fits how your agent runs — both write to the same persistent backend and share the same quality gates.

Path A — @lk.learn (model path)

For any LLM agent. Distilled skills and failures are injected as text context. No tool instrumentation needed.

@lk.learn(domain="legal")
def agent(task, _learnkit_context=""):
    # distilled context injected on every call
    return llm.run(task, system=_learnkit_context)

Path B — @lk.agent_learn (tool-agent path)

For agents that call tools. Captures every tool sequence, replays proven procedures on exact re-encounters, and guides similar tasks — cutting LLM calls by up to 66% across rounds.

@lk.agent_learn(domain="pipeline")
def tool_agent(task, _learnkit_tools=None):
    rows = _learnkit_tools.record("query", args, result)
    # procedure captured ↑  replayed next run
    return rows

Framework adapters

LearnKit is framework-agnostic and ships adapters for popular agent frameworks. Integration typically requires only a few lines using the LearnKit orchestrator class.

  • LangChain — wrap create_agent() / agent.invoke(); context is spliced in via _learnkit_context.
  • LangGraph — node-level memory injection across graph state.
  • AutoGen — distillation around multi-agent conversations.
  • OpenAI — direct integration for the OpenAI SDK.

Dashboard & API

The observability dashboard inspects memory records, retrieval quality, task history, traces, and the memory lifecycle. It talks to the FastAPI server in Docs/server.py and falls back to local mock data when the backend is offline.

EndpointPurpose
GET /healthzLiveness check and loaded playground stores.
GET /api/domainsAvailable playground domains and record counts.
POST /api/inspectRun classify → retrieve → compose and return the injected context.
GET /api/v1/recordsMemory records and related lifecycle routes.

Run locally: start the backend with python Docs/server.py, then run npm run dev in Docs/dashboard — the dev server proxies /api to the backend automatically.
