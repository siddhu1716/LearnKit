Per-model interpretation
========================================================================

qwen2.5-7b (Qwen/Qwen2.5-7B-Instruct)
  - Full PASS on all three suite benchmarks and the playbook regression gate.
  - 62% LLM-call reduction on react_live (21 -> 8) with zero loss in success.
  - 66% LLM-call reduction on evolution_live (58 -> 20) with all 16/16 tasks
    still solved cold and warmed.
  - Playbook injection lifts compliance from 0.375 -> 3.0 (effect +2.625) and
    perfect pass^k=1.0 on novel sibling tasks.
  - Recommendation: this is the reference model for the production playbook
    regression gate today.

qwen2.5-14b (Qwen/Qwen2.5-14B-Instruct)
  - Endpoint reachable; framework end-to-end completed all three benchmarks.
  - Clear LearnKit gain visible:
      react_live success     3/6 -> 5/6  (warmed)
      evolution_live success 10/16 -> 15/16 (warmed, evolved=true)
  - injection_ablation scored 0 across all arms because this endpoint's
    tool-call parser does not consistently extract multi-call hermes-style
    output (the 14B emits <tool_call>...</tool_call> blocks in `content`
    rather than the structured `tool_calls` field). The grader keys on actual
    tool arguments, so missed parses look like zero compliance.
  - Action item (infra, not framework): start the 14B sglang server with
    `--tool-call-parser hermes` and confirm structured tool_calls are
    returned. Once the parser config is corrected, re-run the suite; the
    react/evolution lift indicates the playbook is being applied correctly,
    only the verifier is starved of tool-arg evidence.

deepseek-coder-33b (deepseek-ai/deepseek-coder-33b-instruct)
  - Endpoint reachable; model does NOT emit tool calls. A direct probe
    against /v1/chat/completions with tools=TOOLS_SCHEMA, tool_choice='auto'
    returned a Python code block in `content` and `tool_calls=[]`.
  - This is a model capability gap (the base 33B instruct release predates
    standardized tool-calling). The framework cannot warm a model that never
    produces tool_calls, so cold and warmed are both zero.
  - Recommendation: replace with a coder model that supports structured
    tool-calling (e.g. deepseek-coder-v2-instruct or qwen2.5-coder via the
    hermes parser) if you want a coder lane in the matrix.

========================================================================
Framework portability finding
========================================================================
- The agentic suite, suite orchestrator, and matrix runner now treat every
  model uniformly: same gates, same JSON artifacts, same regression metric.
- Three observed failure classes are now distinguishable in artifacts:
    1) GATE PASS              -> playbook_effect >= threshold (qwen2.5-7b)
    2) PARSER / HARNESS GAP   -> framework executes, but the endpoint's
                                 tool-call parser drops multi-call output
                                 (qwen2.5-14b on injection_ablation)
    3) MODEL CAPABILITY GAP   -> model does not produce tool_calls at all
                                 (deepseek-coder-33b-instruct)
- This separation lets us promote a model into the gating lane only when
  it falls into class (1), while keeping classes (2) and (3) visible in
  reports without blocking the build.

- Keep qwen2.5-14b in the comparative lane; promote it to the gating lane
